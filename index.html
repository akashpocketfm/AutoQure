What The System Now Does

This is no longer just test generation. The current project has a real source-to-device QA pipeline with feature promotion, reporting, and guarded monetization coverage.

1

Explore Source and Layouts

App Explorer and feature analysis scan Kotlin, layout XMLs, navigation paths, and visible UI IDs to discover feature surfaces and generate screen-level specs.

2

Generate Candidate Coverage

Specs are converted into candidate test pools. Today the project has 50 generated cases each across search, episode_nav, wallet, payments, profile, social, unlock, downloads, and adflow.

3

Refine Into Real Page Objects and Tests

High-value features are translated into real page objects, refined Java Appium tests, and execution-specific helpers such as ContentResolver and TestBootstrap.

4

Execute On Real Devices

The runner executes against physical Samsung devices, including automated UiAutomator2 recovery for the Samsung M12 instability that showed up during scale-up.

5

Protect Sensitive Flows

Payment is isolated into a separate seeded sandbox flow, while other seeded capabilities like profile run through explicit account-state handling. This is the project pattern for future monetization and unlock coverage.

Current Blockers and Honest Constraints

The project is broad now, but not every implemented surface is promotion-ready. These are the real remaining constraints.

Runtime / Suite Drift

  • The suite structure has moved from older guest/demo/authenticated naming into a newer trusted plus payment split, and the reporting layer still needs cleanup to match that model fully.
  • Some smoke report files are stale or misleading, so suite truth currently comes more from the test classes and suite XMLs than from the latest generated report file.
  • Blocked features are mostly content-fixture problems now, not missing framework problems.

Fixture / Account Needs

  • Unlock needs deterministic locked content and likely dedicated seeded unlock accounts.
  • Social needs a show or episode where the comments surface is reliably present.
  • Payments must stay isolated to the sandbox payment user and should not be treated as generic smoke for arbitrary accounts.

What This Means Right Now

AutoQure has already moved into practical app-verification territory. The next stage is not proving the concept; it is widening breadth safely and then deepening regression coverage within each feature.

Current Demo Story

We can now demo real feature-based Android coverage across Search, Episode Navigation, Video Player, Wallet, Profile, Library, Language, and sandbox Payments, all backed by concrete Appium classes and suite XMLs rather than only generated ideas.

Next Expansion Logic

The fastest path from here is to keep broadening with deterministic features, while separately solving content fixtures for blocked surfaces like Unlock and Social and promoting deeper regression coverage later.
