The Data Flow Pipeline

Here is exactly how the orchestrator processes a mobile feature.

1

Explore

The App Explorer scans Pocket Toons' raw Android source files. It reads `.kt` files and layout `.xml`, building a deep structural map and identifying available user interactions.

2

Generate

The PRD Parser structures JSON test specifications. Driven by these specs, Claude creates production-oriented Java Appium code targeting our UI elements. (Includes Confidence Scoring: every method is evaluated 0.0-1.0 to flag uncertainties).

3

Compile

Our Orchestrator pauses to ensure safety. It runs a background build via Maven (`mvn -q -DskipTests test-compile`) to validate that the generated Java test code is compile-safe before execution.

4

Execute

Using our Master Runner script, AutoQure wipes device state, establishes logcat recording, and runs the entire compiled execution suite natively within a physical Samsung device lab.

5

Self-Healing

The key capability. If a test fails, the agent intercepts the failure. It classifies the exact error type (UI change, crash, timeout), automatically writes a code patch, rebuilds the APK if necessary, and retries automatically within guarded workflows.

Pipeline Differentiator: Self-Healing

Our autonomous pipeline does more than report bugs—it can generate and validate targeted patches.

A Closed-Loop Defect System

When a traditional pipeline fails, it creates a ticket and blocks delivery. When AutoQure fails, it immediately activates the Self-Healing CLI.

  • Test Fix Engine: Rewrites incorrect selectors, adjusts timeouts, or updates scrolling logic.
  • Crash Analyzer: If the app actually threw an Exception, the AI inspects the Kotlin code and proposes the fix.
  • Sandbox Validation: Rebuilds the artifact and runs a verification step in a closed environment before declaring success.
