Self-Healing and Guarded Autofix Pipeline

This is one of the most important parts of the project story. AutoQure does not just run tests; it diagnoses failures, proposes fixes, and validates them under guardrails before keeping any change.

1

Failure Classification

When a test fails, the Appium healing pipeline classifies the failure category using logs, XML reports, and execution context. Typical classes include timeout, assertion failure, screen-state mismatch, and crash-linked failures.

2

Target Selection

The pipeline maps the failure to the most relevant file and surface, such as a generated test, a page object, or a shared framework helper, instead of blindly editing unrelated code.

3

Suggested or Applied Fix

In suggestion mode, AutoQure records the diagnosis without editing files. In guarded autofix mode, the LLM can generate a constrained candidate patch for the failing Appium code path.

4

Sandbox Validation

Candidate fixes are validated through compile plus targeted rerun workflows using the sandbox executor. If validation fails, touched files are restored instead of leaving the workspace in a broken state.

5

Guardrails and Protected Zones

Protected files and sensitive surfaces such as auth, payment, encryption, credentials, and other high-risk code paths are blocked from unsafe autofix, which keeps healing scoped and safer for production-adjacent work.

Current Blockers and Honest Constraints

The system is broad now, but not every implemented surface is promotion-ready. These are the real remaining constraints.

Coverage / Fixture Gaps

  • Unlock needs deterministic locked content and likely dedicated seeded unlock accounts.
  • Social needs a show or episode where the comments surface is reliably present.
  • Show Detail still needs a consistent, user-reachable navigation path before promotion.

Operational Cleanup Still Needed

  • The suite structure has moved from older guest and demo naming into a newer trusted plus payment split, and the reporting layer still needs final cleanup to align fully.
  • Some generated smoke report files are stale or misleading, so the current source of truth is the suite XML plus the promoted test classes rather than the last report artifact alone.
  • Payment must remain isolated to the sandbox-marked seeded payment account and should not be treated as a normal generic smoke user flow.
