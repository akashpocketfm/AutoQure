The Automation Paradigm Shift

AutoQure is designed to move QA away from slow, brittle feature-by-feature scripting toward guarded AI-assisted coverage generation and recovery.

Standard Automation

  • Engineers manually write Appium tests flow by flow.
  • Minor UI changes can break selectors and destabilize suites.
  • Expanding coverage to new features takes significant manual effort.
  • Failures still require humans to investigate before any fix can begin.

AutoQure System

  • AI generates coverage candidates directly from specs and source code.
  • Generated output passes through quality, compile, and isolation gates before execution.
  • A validated smoke pack now supports controlled build and patch validation on real devices.
  • The execution pipeline classifies failures and can apply targeted fixes through guarded workflows.

The System We’ve Built

Intelligence, execution, and healing are now connected in a single QA pipeline.

Claude-Powered Intelligence

The system integrates with Claude to interpret source, layouts, and feature behavior, reducing manual scripting effort during coverage expansion.

Architecture Deep Dive

AutoQure is now a layered QA system with generation, validation, smoke execution, and guarded recovery working together.

1. Intelligence Layer

  • App Explorer scans Android source and layout XML.
  • PRD Parser converts specs into structured test cases.
  • AI Code Generator creates Java Appium tests with confidence-scored methods.

2. Orchestration Layer

  • run_autoqure_cycle.py manages exploration, generation, validation, execution, and reporting.
  • run_smoke_validation.py executes the validated smoke baseline for internal build and patch checks.
  • Generated output passes through quality, compile, and isolation checks before it becomes runnable.

3. Execution Layer

  • Appium + TestNG suites execute on physical Android devices using warm and cold state strategies.
  • TestBootstrap provides shared overlay dismissal, feed recovery, and start-state handling across validated suites.
  • Execution logs, suite output, and feature-level reports are retained for traceability.

4. Healing Layer

  • Failures are classified by type.
  • Targeted fixes can be generated for selectors, waits, or app code paths.
  • APK rebuild and retry are handled within guarded workflows.

Reliability Guardrails

Recent work focused on making the AI pipeline trustworthy, not just productive.

Quality Gate

  • Low-trust generated classes are screened before execution.
  • TODO markers and placeholder-heavy outputs are rejected before they enter runnable suites.

Compile Isolation

  • Generated classes must pass Maven test-compile before device execution.
  • Excluded generated tests now live outside the Maven source tree, so broken generated code cannot poison validated suites.

Shared Bootstrap

  • Overlay dismissal and feed recovery are now centralized in TestBootstrap instead of duplicated across validated tests.
  • This keeps smoke behavior consistent and reduces drift as new features are hardened.

Execution Modes & Reporting

  • Warm-state execution is now the default for smoke, with cold-state onboarding available as an explicit mode.
  • Per-feature reporting keeps generated, runnable, and device-validated coverage measurable.

Scale-Up Roadmap

The current strategy is controlled expansion, not noisy broad generation.

Stage 1: Operational Baseline

  • Search, episode navigation, and video player are device-validated
  • A trusted smoke baseline now exists for real build and patch validation

Stage 2: Hardening Now

  • Profile is the next candidate feature because it is reachable, high-value, and not content-type dependent
  • Warm-state execution remains the default path for fast, repeatable smoke validation

Stage 3: Controlled Expansion

  • Unlock follows when seeded account and config support are ready
  • Broader features follow only when they pass through the same guarded validation pipeline
  • Explorer-style automation comes later as a scale accelerator, not before adoption is solid
